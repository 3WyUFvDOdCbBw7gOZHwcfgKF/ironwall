# Ironwall 设计纲领

本文不是语法手册，也不是具体工程方案说明，而是 Ironwall 的设计原则与立场声明。它回答的不是「这个语法怎么写」，而是「为什么 Ironwall 要这样设计」。

## 1. 核心立场

Ironwall 的首要目标不是增加复杂语法，不是追求最少键击数，也不是为了 benchmark 上多几个百分点而放松安全边界。Ironwall 的核心立场有三条：

- 安全压倒性优先于性能话术。
- 降低犯错机率优先于增加语言魔法。
- 降低系统复杂度优先于追求理论上的极限优化。

换句话说，Ironwall 不把「相信程序员永远小心」「相信编译器能预知所有运行时状态」「相信更复杂的系统最终会更安全」当成设计前提。

## 2. 记忆体安全立场

### 2.1 强制 GC

Ironwall 明确采取强制垃圾回收路线，并把这条路线视为记忆体安全模型的一部分，而不是可有可无的 runtime 插件。

其理由是：

- 一旦系统规模上升，手动记忆体管理的心智负担会快速失控。
- 把记忆体安全建立在程序员自律、程式码审查、或局部习惯之上，工程上不可靠。
- 为了避免 GC 而引入更多复杂协议，往往只是把风险从表面移到更深层。

因此，Ironwall 不把「无 GC 也能保证记忆体安全」视为语言目标。

Ironwall 在这个问题上的立场不是温和保留，而是明确断言：

- 静态检查不可能保证内存安全。
- 唯一能在语言层与 runtime 层真正保证内存安全的总体方案，只有强制 GC。

这不是口味差异，而是对问题性质的判断。内存安全最终取决于真实运行时的对象生命期、别名关系、可达性、控制流与资料状态；任何宣称只靠静态检查就能彻底保证内存安全的说法，本质上都是在妄想预测未来的运行时资料。

Ironwall 对此的结论非常直接：

- 把静态分析当成内存安全的最终保证，本质上是一种巫术式幻想。
- 它假设编译器能靠超能力在编译期完整预知未来执行期的真实状态。
- 这种前提不可信，因此不能作为 Ironwall 的安全基石。

静态分析可以提供诊断、约束、保守收束，但不能替代 GC，也不能被包装成「已经从根本上解决内存安全」。

### 2.2 性能不是记忆体模型的最高裁判

Ironwall 承认性能重要，但拒绝让性能叙事主导记忆体安全决策。

- 如果某个设计只是为了缩短停顿，却引入更大的可信边界与更高复杂度，那不是进步。
- `zero-cost abstraction` 不是 Ironwall 的价值承诺。
- 抽象几乎总有成本；真正重要的是成本是否透明、可控、可审计。

Ironwall 宁可接受明确成本，也不接受靠语言宣传掩盖真实代价。

## 3. 安全模型立场

### 3.1 不信任人类自律

Ironwall 的规则设计不以「高手会小心」「团队会自律」「大家注意一点就好」为前提。

这不是对个人能力的否定，而是对大型系统工程现实的承认：

- 人会疲劳。
- 团队会流动。
- 规模会扩张。
- 上下文会遗失。

因此，能靠语言规则强制的，就不该退回到「靠纪律」。

### 3.2 静态分析的边界

Ironwall 接受保守、直接、可解释的静态分析，但只把它当作辅助工具，绝不把它提升成记忆体安全的最终宪法。

Ironwall 的立场是：

- 静态分析可以提供边界收束与提早诊断。
- 静态分析不能保证内存安全。
- 真正关键的内存安全必须依赖更硬的 runtime 机制，而不是依赖编译期预言。
- 能靠简化模型与强制 GC 解决的问题，不应优先靠更复杂的分析框架解决。

在这个问题上，Ironwall 明确反对任何把记忆体安全建立在「编译器足够聪明就能完整看穿未来」之上的设计哲学。

### 3.3 Fail-Fast

当系统进入违反前提、型别不一致、tag 非法、越界、或其他核心不变式破坏的状态时，Ironwall 的立场是直接失败，而不是试图用语言级补救机制掩盖问题。

因此：

- Ironwall 不提供语言级 `throw` / `try` / `catch`。
- 可恢复失败应以显式资料模型建模，例如 `union` / `match`。
- 不可恢复失败则应立即中止，而不是拖进更不一致的状态。

## 4. 型别系统立场

### 4.1 必须是强型别语言

Ironwall 明确拒绝动态型别作为核心语言路线。

原因不是「动态型别写不出程式」，而是：

- 早期少打几个字，通常只是把复杂度推迟到后期。
- 程式越大，型别边界越需要明确。
- 若语言鼓励把不确定性延后到执行期，除错成本只会越来越高。

因此，Ironwall 把清楚的型别边界视为工程可维护性的基本要求。

### 4.2 显式优先

Ironwall 不把「少打字」当成第一原则。

所以：

- 绑定必须显式型别。
- 函数参数与返回型别必须显式书写。
- 泛型实例化必须显式提供型别参数。
- 不提供数值预设型别。

型别推导若会模糊语义、隐藏选择点、或让错误讯息退化，就不是 Ironwall 想要的能力。

### 4.3 不接受 `null`

Ironwall 拒绝把 `null` 当作通用逃生口。

缺失值应显式建模，而不是用一个可以渗透整个系统的特殊空值污染所有 API。这也是为什么 Ironwall 接受 `union` / `match`，因为它们让「可能不存在」变成型别层可见的事实，而不是隐蔽约定。

### 4.4 保留必要抽象，不追求抽象泛滥

Ironwall 不反抽象，但只接受成本与收益都清楚的抽象。

明确支持且认为值得保留的能力包括：

- 高阶函数
- 简单泛型
- `union` 与 `match`

这些能力之所以被接受，不是因为它们「理论上高级」，而是因为它们能在不引爆系统复杂度的前提下，实际提升表达力与安全性。

## 5. 语法与可读性立场

### 5.1 简单语法优先

Ironwall 明确偏好简单、统一、结构稳定的语法，而不偏好多套糖层叠加的表面易读性。

因此：

- 核心语法使用简单的 S-expression 家族结构。
- 语法种类尽量少，括号类型与语义域明确分离。
- 不鼓励引入大量特例写法。

### 5.2 可读性可以由工具补强

Ironwall 承认类 S-expression 语法在纯文本下未必是最直观的阅读形式，因此把部分可读性工作交给工具层，例如：

- AST visualization
- structured editor
- graphical syntax projection

这代表 Ironwall 的立场是：

- 语言核心先保持规则单纯。
- 更高层的人机介面可以由 IDE 负责补强。

## 6. 复杂度立场

Ironwall 对复杂度高度不信任。

它不相信以下路线天然更安全：

- 更复杂的语义角落
- 更复杂的记忆体模型
- 更复杂的宏系统
- 更复杂的并发保证
- 更复杂的补救机制

Ironwall 的基本判断是：复杂度越高，可信边界越模糊，审计成本越高，真正出事时越难定位。

因此，若两条路都能达到相近效果，Ironwall 会优先选择：

- 规则更少的
- 更容易解释的
- 更容易验证的
- 对实作者与使用者都更诚实的

## 7. 对宏与元编程的立场

Ironwall 彻底拒绝宏系统与预处理器。

理由很直接：

- 宏会破坏「看到什么就是什么」。
- 宏会让语法分析、工具链、审计、诊断都变复杂。
- 宏经常把局部便利转化为全域不可预测性。

Ironwall 不认为这是值得交换的代价。

## 8. 对运行时检查的立场

Ironwall 不排斥 runtime checks，反而视其为安全模型的一部分。

因此：

- 阵列存取必须做边界检查。
- `if` / `while` 条件必须是 `bool`，不接受把整数当布林值使用。
- 非法 tag、越界、违反核心前提等情况都应立即失败。

这些检查不是语言「不够高级」的表现，而是对真实风险保持诚实。

## 9. 对并发的立场

Ironwall 不把「语言层自动保证并发安全」作为设计目标。

其原因是：

- 并发正确性高度依赖具体运行时资料与调度条件。
- 若在语言层宣称能一劳永逸处理并发安全，往往只会制造错误期待。

因此，Ironwall 优先把单执行绪、型别、模组、初始化、记忆体与失败模型收束到位，而不是提前承诺超出可信范围的并发语义。

## 10. 反目标

以下事项明确不是 Ironwall 的目标：

- 为了少打几个字而牺牲边界清晰
- 为了性能宣传而放松记忆体安全模型
- 为了语法华丽而引入大量特例
- 为了抽象自由度而放任宏与隐式行为
- 为了看起来先进而引入高复杂度 GC 协议
- 为了局部便利而把整体语言做成难以审计的系统

## 11. 总结

Ironwall 的设计信条可以浓缩为一句话：

> 用更硬的语言边界、更少的复杂度、和更诚实的 runtime 成本，换取更可信的安全性与可维护性。
