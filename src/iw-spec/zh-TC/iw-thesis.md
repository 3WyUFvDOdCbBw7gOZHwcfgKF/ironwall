# Ironwall 設計綱領

本文不是語法手冊，也不是具體工程方案說明，而是 Ironwall 的設計原則與立場聲明。它回答的不是「這個語法怎麼寫」，而是「為什麼 Ironwall 要這樣設計」。

## 1. 核心立場

Ironwall 的首要目標不是增加複雜語法，不是追求最少鍵擊數，也不是為了 benchmark 上多幾個百分點而放鬆安全邊界。Ironwall 的核心立場有三條：

- 安全壓倒性優先於性能話術。
- 降低犯錯機率優先於增加語言魔法。
- 降低系統複雜度優先於追求理論上的極限優化。

換句話說，Ironwall 不把「相信程序員永遠小心」「相信編譯器能預知所有運行時狀態」「相信更複雜的系統最終會更安全」當成設計前提。

## 2. 記憶體安全立場

### 2.1 強制 GC

Ironwall 明確採取強制垃圾回收路線，並把這條路線視為記憶體安全模型的一部分，而不是可有可無的 runtime 插件。

其理由是：

- 一旦系統規模上升，手動記憶體管理的心智負擔會快速失控。
- 把記憶體安全建立在程序員自律、程式碼審查、或局部習慣之上，工程上不可靠。
- 為了避免 GC 而引入更多複雜協議，往往只是把風險從表面移到更深層。

因此，Ironwall 不把「無 GC 也能保證記憶體安全」視為語言目標。

Ironwall 在這個問題上的立場不是溫和保留，而是明確斷言：

- 靜態檢查不可能保證內存安全。
- 唯一能在語言層與 runtime 層真正保證內存安全的總體方案，只有強制 GC。

這不是口味差異，而是對問題性質的判斷。內存安全最終取決於真實運行時的對象生命期、別名關係、可達性、控制流與資料狀態；任何宣稱只靠靜態檢查就能徹底保證內存安全的說法，本質上都是在妄想預測未來的運行時資料。

Ironwall 對此的結論非常直接：

- 把靜態分析當成內存安全的最終保證，本質上是一種巫術式幻想。
- 它假設編譯器能靠超能力在編譯期完整預知未來執行期的真實狀態。
- 這種前提不可信，因此不能作為 Ironwall 的安全基石。

靜態分析可以提供診斷、約束、保守收束，但不能替代 GC，也不能被包裝成「已經從根本上解決內存安全」。

### 2.2 性能不是記憶體模型的最高裁判

Ironwall 承認性能重要，但拒絕讓性能敘事主導記憶體安全決策。

- 如果某個設計只是為了縮短停頓，卻引入更大的可信邊界與更高複雜度，那不是進步。
- `zero-cost abstraction` 不是 Ironwall 的價值承諾。
- 抽象幾乎總有成本；真正重要的是成本是否透明、可控、可審計。

Ironwall 寧可接受明確成本，也不接受靠語言宣傳掩蓋真實代價。

## 3. 安全模型立場

### 3.1 不信任人類自律

Ironwall 的規則設計不以「高手會小心」「團隊會自律」「大家注意一點就好」為前提。

這不是對個人能力的否定，而是對大型系統工程現實的承認：

- 人會疲勞。
- 團隊會流動。
- 規模會擴張。
- 上下文會遺失。

因此，能靠語言規則強制的，就不該退回到「靠紀律」。

### 3.2 靜態分析的邊界

Ironwall 接受保守、直接、可解釋的靜態分析，但只把它當作輔助工具，絕不把它提升成記憶體安全的最終憲法。

Ironwall 的立場是：

- 靜態分析可以提供邊界收束與提早診斷。
- 靜態分析不能保證內存安全。
- 真正關鍵的內存安全必須依賴更硬的 runtime 機制，而不是依賴編譯期預言。
- 能靠簡化模型與強制 GC 解決的問題，不應優先靠更複雜的分析框架解決。

在這個問題上，Ironwall 明確反對任何把記憶體安全建立在「編譯器足夠聰明就能完整看穿未來」之上的設計哲學。

### 3.3 Fail-Fast

當系統進入違反前提、型別不一致、tag 非法、越界、或其他核心不變式破壞的狀態時，Ironwall 的立場是直接失敗，而不是試圖用語言級補救機制掩蓋問題。

因此：

- Ironwall 不提供語言級 `throw` / `try` / `catch`。
- 可恢復失敗應以顯式資料模型建模，例如 `union` / `match`。
- 不可恢復失敗則應立即中止，而不是拖進更不一致的狀態。

## 4. 型別系統立場

### 4.1 必須是強型別語言

Ironwall 明確拒絕動態型別作為核心語言路線。

原因不是「動態型別寫不出程式」，而是：

- 早期少打幾個字，通常只是把複雜度推遲到後期。
- 程式越大，型別邊界越需要明確。
- 若語言鼓勵把不確定性延後到執行期，除錯成本只會越來越高。

因此，Ironwall 把清楚的型別邊界視為工程可維護性的基本要求。

### 4.2 顯式優先

Ironwall 不把「少打字」當成第一原則。

所以：

- 綁定必須顯式型別。
- 函數參數與返回型別必須顯式書寫。
- 泛型實例化必須顯式提供型別參數。
- 不提供數值預設型別。

型別推導若會模糊語義、隱藏選擇點、或讓錯誤訊息退化，就不是 Ironwall 想要的能力。

### 4.3 不接受 `null`

Ironwall 拒絕把 `null` 當作通用逃生口。

缺失值應顯式建模，而不是用一個可以滲透整個系統的特殊空值污染所有 API。這也是為什麼 Ironwall 接受 `union` / `match`，因為它們讓「可能不存在」變成型別層可見的事實，而不是隱蔽約定。

### 4.4 保留必要抽象，不追求抽象泛濫

Ironwall 不反抽象，但只接受成本與收益都清楚的抽象。

明確支持且認為值得保留的能力包括：

- 高階函數
- 簡單泛型
- `union` 與 `match`

這些能力之所以被接受，不是因為它們「理論上高級」，而是因為它們能在不引爆系統複雜度的前提下，實際提升表達力與安全性。

## 5. 語法與可讀性立場

### 5.1 簡單語法優先

Ironwall 明確偏好簡單、統一、結構穩定的語法，而不偏好多套糖層疊加的表面易讀性。

因此：

- 核心語法使用簡單的 S-expression 家族結構。
- 語法種類儘量少，括號類型與語義域明確分離。
- 不鼓勵引入大量特例寫法。

### 5.2 可讀性可以由工具補強

Ironwall 承認類 S-expression 語法在純文本下未必是最直觀的閱讀形式，因此把部分可讀性工作交給工具層，例如：

- AST visualization
- structured editor
- graphical syntax projection

這代表 Ironwall 的立場是：

- 語言核心先保持規則單純。
- 更高層的人機介面可以由 IDE 負責補強。

## 6. 複雜度立場

Ironwall 對複雜度高度不信任。

它不相信以下路線天然更安全：

- 更複雜的語義角落
- 更複雜的記憶體模型
- 更複雜的宏系統
- 更複雜的並發保證
- 更複雜的補救機制

Ironwall 的基本判斷是：複雜度越高，可信邊界越模糊，審計成本越高，真正出事時越難定位。

因此，若兩條路都能達到相近效果，Ironwall 會優先選擇：

- 規則更少的
- 更容易解釋的
- 更容易驗證的
- 對實作者與使用者都更誠實的

## 7. 對宏與元編程的立場

Ironwall 徹底拒絕宏系統與預處理器。

理由很直接：

- 宏會破壞「看到什麼就是什麼」。
- 宏會讓語法分析、工具鏈、審計、診斷都變複雜。
- 宏經常把局部便利轉化為全域不可預測性。

Ironwall 不認為這是值得交換的代價。

## 8. 對運行時檢查的立場

Ironwall 不排斥 runtime checks，反而視其為安全模型的一部分。

因此：

- 陣列存取必須做邊界檢查。
- `if` / `while` 條件必須是 `bool`，不接受把整數當布林值使用。
- 非法 tag、越界、違反核心前提等情況都應立即失敗。

這些檢查不是語言「不夠高級」的表現，而是對真實風險保持誠實。

## 9. 對並發的立場

Ironwall 不把「語言層自動保證並發安全」作為設計目標。

其原因是：

- 並發正確性高度依賴具體運行時資料與調度條件。
- 若在語言層宣稱能一勞永逸處理並發安全，往往只會製造錯誤期待。

因此，Ironwall 優先把單執行緒、型別、模組、初始化、記憶體與失敗模型收束到位，而不是提前承諾超出可信範圍的並發語義。

## 10. 反目標

以下事項明確不是 Ironwall 的目標：

- 為了少打幾個字而犧牲邊界清晰
- 為了性能宣傳而放鬆記憶體安全模型
- 為了語法華麗而引入大量特例
- 為了抽象自由度而放任宏與隱式行為
- 為了看起來先進而引入高複雜度 GC 協議
- 為了局部便利而把整體語言做成難以審計的系統

## 11. 總結

Ironwall 的設計信條可以濃縮為一句話：

> 用更硬的語言邊界、更少的複雜度、和更誠實的 runtime 成本，換取更可信的安全性與可維護性。
